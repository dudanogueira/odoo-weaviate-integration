O que Acontece ao Instalar um App

Sequência de instalação:
  • Lê o __manifest__.py
  • Instala dependências (depends)
  • Executa models → cria tabelas no PG
  • Carrega views e security
  • Roda arquivos de data e demo
  • Registra rotas dos controllers
De onde vêm os módulos:
  • Pasta local no addons_path
  • Odoo Apps Store (cloud)
  • Upload de arquivo .zip
  • Git + symlink (desenvolvimento)
Atualizar um módulo:

Apps → filtrar "Instalados" → Atualizar
ou via CLI: -u nome_modulo

_inherit é o poder central do Odoo: adiciona campos e sobrescreve métodos sem tocar no código original. O modelo original continua intacto e recebe atualizações normalmente.

Views & QWeb: Herança XML

View original (módulo product)
<form id="product_template_form_view">
  <sheet>
    <group>
      <field name="name"/>
      <field name="list_price"/>
    </group>
    <!-- ... -->
  </sheet>
</form>
Injetando nosso campo via herança
<record model="ir.ui.view">
  <!-- Referencia a view original -->
  <field name="inherit_id" ref=
    "product.product_template_form_view"/>

  <field name="arch" type="xml">
    <!-- XPath: onde injetar -->
    <xpath expr="//field[@name='name']"
           position="after">
      <field name="weaviate_uuid"
             readonly="1"/>
    </xpath>
  </field>
</record>
💡 A view original não é modificada. O Odoo mescla as heranças em runtime.

Controllers: Herança de Rotas HTTP

from odoo import http
from odoo.addons.website_sale.controllers.main import WebsiteSale


class WebsiteSaleWeaviate(WebsiteSale):
    """
    Herda o controller principal do e-commerce.
    O Odoo descobre e usa esta classe automaticamente
    no lugar da original — sem precisar de registro.
    """

    @http.route()          # mantém a rota /shop original
    def shop(self, page=0, category=None, search='', **post):
        # O model já intercepta em _search() via ORM.
        # O controller garante a ordenação correta na página.
        return super().shop(
            page=page, category=category, search=search, **post
        )
Como o Odoo descobre o controller herdado?
O framework usa Python MRO (Method Resolution Order). Ao carregar os módulos instalados, o Odoo constrói a hierarquia de classes automaticamente.

Bancos de Dados
Vetoriais

O Problema da Busca Tradicional

SQL ILIKE 
SELECT * FROM product_template
WHERE name ILIKE '%tenis%'
  • ❌ Não acha "tênis" (acento)
  • ❌ Não acha "calçado esportivo"
  • ❌ Não entende sinônimos
  • ❌ Não entende contexto
  • ❌ Relevância zero — tudo igual
Busca Semântica 
weaviate.hybrid_search(
    "tênis corrida",
    alpha=0.5
)
  • ✅ Ignora variações de escrita
  • ✅ Entende sinônimos
  • ✅ Busca por significado
  • ✅ Resultados por relevância
  • ✅ Combina com keyword search

O que são Embeddings?

Texto → Vetor Numérico

Um modelo de IA converte qualquer texto em um array de ~1536 números que captura o significado semântico da frase.

"tênis de corrida"  → [0.23, -0.45, 0.87, ...]
"calçado esportivo" → [0.21, -0.44, 0.85, ...]
"notebook Dell"     → [-0.67, 0.12, -0.34, ...]
Textos semanticamente similares ficam próximos no espaço vetorial

Hybrid Search

O melhor dos dois mundos: keyword + semântica

🔤
BM25

Keyword search clássico. Ótimo para termos exatos, SKUs, códigos.

⚖️
alpha = 0.5
RELATIVE_SCORE fusion
🧠
Vetor

Busca semântica. Ótimo para linguagem natural e sinônimos.

def hybrid_search(self, query: str, limit: int = 50, alpha: float = 0.5):
    with self._get_client() as client:
        collection = client.collections.get(self.collection_name)
        results = collection.query.hybrid(
            query=query,
            alpha=alpha,              # 0 = só BM25 · 1 = só vetor
            limit=limit,
            fusion_type=HybridFusion.RELATIVE_SCORE,
            return_metadata=MetadataQuery(score=True),
        )
        # Retorna {odoo_id: score} ordenado por relevância
        return {int(o.properties["odoo_id"]): o.metadata.score
                for o in results.objects}

O que é o Weaviate?

🗃️
Banco Vetorial Open Source

Módulos de Vetorização

product_weaviate_search

O que o módulo faz — sem alterar uma linha do Odoo core

1
Sincronização em tempo real

Hooks nos eventos create/write/unlink mantêm o Weaviate atualizado

2
Busca backend substituída

Lista de produtos usa hybrid search em vez de SQL ILIKE

3
Busca do e-commerce substituída

Rota /shop interceptada via herança de controller

4
Wizard de sincronização em massa

Re-indexa todos os produtos, com reset e relatório de erros

Tolerância a falhas: Se o Weaviate estiver fora, o Odoo continua com a busca SQL. Nenhuma operação de salvar/editar produto é bloqueada.

Sincronização em Tempo Real

class ProductTemplate(models.Model):
    _inherit = 'product.template'

    def create(self, vals_list):
        records = super().create(vals_list)
        if self._weaviate_live_sync_enabled():
            records._weaviate_sync_after_write()
        return records

    def write(self, vals):
        result = super().write(vals)
        # Só sincroniza se campos relevantes foram alterados
        watched = {'name', 'description', 'list_price',
                   'categ_id', 'description_sale', 'active'}
        if self._weaviate_live_sync_enabled() and (watched & vals.keys()):
            self._weaviate_sync_after_write()
        return result

    def unlink(self):
        # Guarda UUIDs antes de deletar
        uuids = self.filtered('weaviate_uuid').mapped('weaviate_uuid')
        result = super().unlink()
        if uuids and self._weaviate_live_sync_enabled():
            service = WeaviateService.from_config(self.env)
            for uuid in uuids:
                service.delete_product(uuid)
        return result
💡 UUID determinístico (UUID5 do odoo_id) garante verdadeiro upsert — sem duplicatas

Override do _search()

def _search(self, domain, offset=0, limit=None, order=None):
    # 1. Extrai o termo de texto do domain do ORM
    query = self._extract_weaviate_query(domain)

    if query and self._weaviate_backend_search_enabled():
        service = WeaviateService.from_config(self.env)
        if service:
            try:
                # 2. Hybrid search → {id: score} ordenado
                scores = service.hybrid_search(query)

                # 3. Guarda scores em thread-local (isolado por request)
                _weaviate_scores.scores = scores

                # 4. Retorna IDs na ordem de relevância
                if scores:
                    return list(scores.keys())
            except Exception as e:
                _logger.warning("Weaviate search failed, fallback: %s", e)

    # 5. Fallback: busca SQL normal
    return super()._search(domain, offset, limit, order)
Thread-local storage
Scores isolados por request. Seguro em ambiente multi-worker do Odoo (Gunicorn).
Fallback automático
Qualquer exceção → cai silenciosamente no SQL ILIKE padrão.

E-commerce Sem Tocar no Código Base

# controllers/website_sale.py
try:
    from odoo.addons.website_sale.controllers.main import WebsiteSale
    WEBSITE_SALE_AVAILABLE = True
except ImportError:
    WEBSITE_SALE_AVAILABLE = False


if WEBSITE_SALE_AVAILABLE:

    class WebsiteSaleWeaviate(WebsiteSale):
        """
        Apenas herdar esta classe já é suficiente.
        O Odoo usa esta versão automaticamente no lugar de
        WebsiteSale — sem registro, sem config, sem tocar no
        módulo website_sale.
        """
        @http.route()
        def shop(self, page=0, category=None, search='', **post):
            # O model intercepta em _search() via ORM.
            # Aqui garantimos a ordenação correta dos resultados.
            return super().shop(
                page=page, category=category, search=search, **post
            )
A mágica: O módulo website_sale é opcional. O import é dentro de try/except. Se o e-commerce não estiver instalado, nada quebra.
